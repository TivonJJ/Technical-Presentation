# 演讲者快速参考

## 核心信息点

### 1. 开场（3 分钟）
**关键信息：**
- AI 技术演进路线：LLM → RAG → Function Calling/MCP → Agent
- 三个核心问题：如何理解、如何知道、如何做事
- 目标：构建真正可用的 AI Agent

**话术建议：**
> "大家好，今天我们要探讨的是大模型如何从一个简单的聊天工具，进化成能够真正执行业务任务的智能体。"

---

### 2. Context & Prompt Engineering（5 分钟）
**关键信息：**
- LLM 无长期记忆，依赖当前窗口
- Context = 模型做判断的全部输入
- 不同模型上下文窗口差异大（Claude 1M vs Grok 128K）

**重点强调：**
- Kiro AI IDE 的对话记录就是上下文的直观展示
- 包含文件、消息、System Prompt 等

---

### 3. RAG（8 分钟）
**关键信息：**
- 痛点：私有文档 AI 不知道，强行回答会幻觉
- 价值：实时补充上下文，降低幻觉，无需重训练
- 缺陷：向量检索不精准，切片丢失结构

**案例：**
> "想象一句话：'我昨天把手机掉进水里了，所以今天不能联系你。'
> 如果切成两个片段，只检索到'今天不能联系你'，模型就不知道原因了。"

---

### 4. GraphRAG（3 分钟）
**关键信息：**
- 传统 RAG：切片 → 嵌入 → 检索（丢失结构）
- GraphRAG：抽取实体 → 构建图谱 → 多跳推理
- 适用场景：复杂文档、技术知识库、系统依赖

**对比重点：**
| 能力 | 传统 RAG | GraphRAG |
|------|---------|----------|
| 理解结构 | ❌ | ✔ |
| 多跳推理 | ❌ | ✔ |

---

### 5. RAG 实际应用（2 分钟）
**关键信息：**
- 适合：知识库、AI 客服、SDK 文档生成
- 不适合：需要明确 Action 的严谨场景
- 演进：Cursor 从 RAG 改用 search/grep，精准度大幅提升

---

### 6. ReAct（3 分钟）
**关键信息：**
- Reasoning + Action：模型自己规划 → 调用工具 → 继续推理
- 缺点：输出格式不稳定，识别率低（小模型）

**流程示例：**
```
Thought → Action → Observation → Thought → Final Answer
```

---

### 7. Function Calling（5 分钟）
**关键信息：**
- 2023.6 OpenAI 推出，标准化工具调用
- 本质：JSON 格式的工具调用指令，由系统执行
- 优势：稳定、安全、可控、开发者友好

**重点：**
- 展示 Request Schema 和 Response 示例
- 强调从"聊天助手"到"任务执行器"的转变

---

### 8. MCP（5 分钟）
**关键信息：**
- Model Context Protocol：AI 世界的 USB 协议
- 核心思想：工具变成 Server，任何 LLM 都能连接
- 三大能力：统一注册、统一资源访问、统一调用规范

**对比表格：**
| 特性 | Function Calling | MCP |
|------|-----------------|-----|
| 本质 | 调用方式 | 接入协议 |
| 位置 | 推理链内 | 工具生态层 |

**关键话术：**
> "它们不是二选一，而是互补：LLM 通过 Function Calling 去调用 MCP Server。"

---

### 9. Agent Skills（3 分钟）
**关键信息：**
- Anthropic 2025.10.16 发布
- 特点：结构化输入输出，不参与上下文，可调用 MCP
- 适用：固定流程的重复性任务

---

### 10. Agent（8 分钟）
**关键信息：**
- 为什么需要：业务任务是多步骤、多工具、多依赖的动态流程
- 价值：持续工作能力，无人工干预完成任务链路

**两种模式：**
1. **ReAct 模式**：灵活，适合探索性任务，但不稳定
2. **Plan & Execute**：先规划后执行，稳定可控，企业级首选

**重点：**
- Cursor、Claude Code 都用 Plan & Execute
- 展示 plan-and-execute.png 图

---

### 11. Multi-Agent（5 分钟）
**关键信息：**
- 多个 Agent 分工协作：检索、写作、审核、格式化
- A2A 协议：Google 2025.4.9 发布，规范 Agent 间通信
- Agentic 应用：大量 Agent "员工"，各司其职

**工具栈：**
- RAG + MCP + GraphRAG + Tool Calling

---

### 12. Agent 开发工具（3 分钟）
**两类工具：**
1. **专业框架**（工程师）：LlamaIndex、LangChain、LangGraph、AutoGen
2. **可视化平台**（产品/业务）：Coze、n8n、Flowise

---

### 13. 总结（5 分钟）
**技术演进路线：**
1. LLM：推理与理解
2. Context：决定理解范围
3. RAG/GraphRAG：知识更新与结构理解
4. Tools/MCP：行动能力
5. Agent：自主规划与执行

**三个核心问题：**
1. 如何理解（Context）
2. 如何知道（RAG）
3. 如何做事（Tools/MCP/Agent）

**结束语：**
> "我们要构建的不是聊天机器人，而是能够获取知识、调用工具、执行任务、自主规划的智能体。"

## 常见问题准备

### Q1: RAG 和 GraphRAG 应该选哪个？
**A:** 取决于场景。简单 FAQ 用传统 RAG，复杂文档、技术知识库用 GraphRAG。

### Q2: MCP 会取代 Function Calling 吗？
**A:** 不会，它们是互补关系。MCP 是工具接入协议，Function Calling 是调用机制。

### Q3: 如何选择 Agent 框架？
**A:** 工程团队用 LangChain/LangGraph，产品团队可以用 Coze 等可视化平台。

### Q4: Agent 的成本如何控制？
**A:** 
- 选择合适的模型（不是所有任务都需要 GPT-4）
- 优化 Context 大小
- 使用缓存机制
- Plan & Execute 模式比 ReAct 更可控

### Q5: 如何评估 Agent 的效果？
**A:**
- 任务完成率
- 执行时间
- 成本
- 错误率
- 用户满意度
